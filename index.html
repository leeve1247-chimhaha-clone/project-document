침하하 클론

저는 구독자 300만 유튜버 침착맨이 운영하는 웹사이트 침하하를 클론 프로젝트로서 진행하고 있습니다. 아마존이나 트위터보다 구현 난이도가 낮으면서도 실제로 프로덕션 단계에서 제대로 운영하고 있어서 실전성도 있다고 느껴졌기 때문입니다. 가장 마지막으로 수정한 웹 애플리케이션의 형태는 다음과 같습니다.

클라우드 대신 로컬을 선택한 이유

AWS 를 쓰지 않은 것은 온디멘드 방식의 금액 산출방식에 불만이었기 때문입니다. 그보다는 한번의 지출로 소요 비용이 예측가능하게 두고 싶어 위와 같은 방식으로 전체 구조를 설계했습니다.

그러나 일반 로컬에서 서비스를 하게 되면 일반적인 포트 포워딩은 방화벽에 구멍을 뚫는 것과 같아 보안 취약점이 될 수 있습니다. 이때 Zero Trust Tunnel을 사용하면 인바운드 포트를 전혀 열지 않고도 외부 접속을 안전하게 처리할 수 있습니다.
적용하는 방법도 Zero Trust 를 설정하면 Cloudflare 다운로드 가능한 개인용 도커 이미지를 컨테이너로 실행하면 되기 때문에 쉽고 간단했습니다.

미니 PC 가 풀스택을 담당하여 내부 컨테이너에서 실행되는 것은 다음과 같습니다.

3. 상세 수행 내용

개인 웹페이지를 프로덕션 레벨로 고도화하면서 중요하게 여긴 요소들을 중심으로 작업을 진행했습니다.

1. 배포 환경 동기화

오랫동안 방치되었던 프로젝트이다 보니 로컬 개발 환경과 실제 배포 환경 간의 괴리가 컸습니다. 초기에는 하나의 docker-compose.yml 파일을 기반으로 환경 변수(.env)만 교체하여 이 문제를 해결하려 했습니다.

하지만 배포 환경을 재구축하는 과정에서, 단일 파일 관리가 오히려 복잡도를 높인다는 사실을 발견했습니다.

  • 보안 및 설정의 차이: URL 스키마, 복잡한 비밀번호 정책, 스토리지 구성 등이 판이하게 달랐습니다.
  • 인프라 아키텍처의 차이: 배포(Prod) 환경은 보안을 위해 Cloudflare Zero Trust(Tunnel)를 통해 외부 트래픽을 처리하지만, 개발(Dev) 환경은 간단한 Nginx 리버스 프록시로 처리합니다.

그래서 개발용과 배포용 파일 및 파라미터를 폴더(리포지토리) 단위로 분리하기로 했습니다.

2. 스토리지 마이그레이션 (MinIO → SeaweedFS)

기존에는 객체 스토리지로 MinIO를 고려했으나, 최근 오픈소스 진영에 적대적인 라이선스 정책 변경을 확인하고 대안이 필요하다고 판단했습니다. 조사한 대안은 다음과 같고, 살펴본 특성은 이렇습니다.

우선 GarageFS 를 사용해보았으나 문서에 설명이 부족하고, 특히 버킷별 정책 설정관련하여 부족한 점이 있어 SeaweedFS 로 마이그레이션을 수행했습니다. OpenMaxIO는 훌륭한 대안이지만 최근 업데이트가 더뎌 유지보수의 지속성을 확신하기 어려웠고, Ceph는 미니 PC 환경에서 구동하기에는 하드웨어 리소스(RAM) 요구사항이 과도했습니다. 그래서 SeaweedFS 로 마이그레이션을 결정하였고, 공식 깃헙과 도커 이미지 설명란을 살펴보면서 적용할 수 있었습니다.

3. 인증 서버 트러블 슈팅

Keycloak 인증 서버의 관리자 계정 콘솔을 브라우저로 접속했을 때, 리다이렉트가 무한히 일어나는 문제가 발생했습니다. Keycloak 컨테이너를 디버그 옵션으로 설정하고 문제를 알아본 뒤, 이는 리버스 프록시 설정에서 포워딩되는 데이터 정보에 차이가 발생해서 일어난 문제라는 것을 알았습니다.

Keycloak 설명서 및 깃헙 이슈란을 살펴보고 컨테이너 실행 환경 변수 중 KC_PROXY_HEADERS , KC_HOSTNAME 를 추가 설정을 해야한다는 사실알게 되어 이를 추가했습니다.

개발 환경에서는 Nginx 의 X-Forwarded 헤더에 XFF (포워딩 IP), XFP (포워딩 TLS 여부), XFPort (포워딩 Port), XRI (실제 접속 시도한 IP) 를 추가함으로써 해결할 수 있었습니다.

배포환경인 Zero Trust 에서는 이러한 포워딩 데이터를 사전 설정없이 기본값으로 제공해주고 있어서 리버스 프록시 설정은 필요 없었습니다. 그래서 KeyCloak 에서 제공하는 배포 시 권장하는 URI 규칙만 설정하여 처리했습니다.

4. 소감

그동안 프로젝트 진행 사항을 정돈하여 글로 남겨놓는 것을 숙제처럼 여기며 자꾸 미뤄왔었는데, 이번 2주간 프리코스를 통해 어질러져 있던 프로젝트를 정돈할 수 있어서 좋았습니다. 단일 프로젝트임에도 불구하고 실제 배포를 할 수 있게끔 구성하면서, 실제 상용단계에서 고민해볼법한 이슈들이 지속적으로 발생하여 꾸준히 다듬고 개선해나가면 백엔드뿐만 아니라 웹 서비스 운영 전반에 대한 이해를 높일 수 있을 것으로 기대가 됩니다. 따라서 앞으로도 이를 지속적으로 수정해나갈 것입니다.

5. 향후 계획

웹 애플리케이션 클론 프로젝트를 수행하던 중, 프론트엔드에서 여러 카테고리의 게시글을 한 페이지에 동시에 게시하고 있는 구조를 발견했습니다. 이를 백엔드에서 요청할 캐싱을 활용하는 게 좋다고 판단하여, Redis를 제대로 적용해 봐야겠다고 느꼈습니다.

지금 상태에서는 간단히 게시글 뷰 카운트(조회수)를 올리는 데에만 쓰고 있는데, 추후 스프링의 @Cacheable 어노테이션을 활용하여 가용성 높은 코드를 작성할 계획입니다.
